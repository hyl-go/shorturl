# 短链服务技术说明（当前实现）

本文档仅描述：**数据库表设计**、**API 接口设计**、**核心业务流程**，以及 **缓存穿透 / 击穿 / 雪崩** 等常见问题在本项目中的处理方式与边界。

---

## 一、数据库表设计

### 1.1 `short_url_map`（长链与短链映射）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | `bigint unsigned` | PK，`AUTO_INCREMENT` | 主键 |
| `create_at` | `datetime` | `NOT NULL`，默认 `CURRENT_TIMESTAMP` | 创建时间 |
| `create_by` | `varchar(64)` | `NOT NULL`，默认 `''` | 创建者 |
| `is_del` | `tinyint unsigned` | `NOT NULL`，默认 `0` | 软删除标记 |
| `lurl` | `varchar(160)` | 可空 | 原始长链接 |
| `md5` | `char(32)` | 可空 | 长链接 MD5，用于幂等查询 |
| `surl` | `varchar(128)` | 可空 | 短链路径段（自定义或 Base62 自动生成） |
| `expire_at` | `datetime` | 可空 | 过期时间；`NULL` 表示永不过期 |

**索引与约束：**

- 主键：`PRIMARY KEY (id)`
- 二级索引：`INDEX (is_del)`（按删除标记过滤）
- 唯一约束：`UNIQUE (lurl)`、`UNIQUE (surl)` —— 同一长链只对应一条记录，短链字符串全局唯一（含自定义短链）

**引擎与字符集：** `InnoDB`，`utf8mb4`。

**设计要点：**

- `surl` 扩至 `varchar(128)`，以容纳自定义别名（不限于原 Base62 发号长度）。
- `expire_at` 与业务层解析（RFC3339）配合，查询跳转时在应用层判断是否在有效期内。

### 1.2 `sequence`（发号器，仅自动生成短链时使用）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `bigint unsigned`，`AUTO_INCREMENT` | 递增序号 |
| `stub` | `varchar(1)`，`UNIQUE` | 固定单行占位（如 `'a'`），`REPLACE INTO` 触发自增 |
| `timestamp` | `timestamp` | 更新时间 |

**引擎：** `MyISAM`（脚本定义）。用于通过 `REPLACE INTO sequence(stub) VALUES('a')` 获取单调递增 `id`，再经 Base62 编码生成默认短链。

## 二、API 接口设计

定义文件：`shorturl/shorturl.api`。对外暴露两个路由。

### 2.1 长链转短链

| 项目 | 说明 |
|------|------|
| **方法与路径** | `POST /convert` |
| **Content-Type** | `application/json` |

**请求体 `ConverRequest`**

| JSON 字段 | 类型 | 必填 | 说明 |
|-----------|------|------|------|
| `longURL` | `string` | 是 | 待缩短的长链接 |
| `customShortURL` | `string` | 否 | 自定义短链字符串；为空则走发号器 + Base62 |
| `expireAt` | `string` | 否 | 过期时间，**RFC3339**，例如 `2026-12-31T23:59:59+08:00` |

**响应体 `ConvertResponse`**

| JSON 字段 | 类型 | 说明 |
|-----------|------|------|
| `shortURL` | `string` | 完整短链（配置中的短链域名 + `/` + `surl`） |
| `expireAt` | `string` | 与请求一致时使用 RFC3339 回显；未设置过期时可为空 |

**成功行为：** HTTP 200，`httpx.OkJson` 返回 JSON。

**失败行为：** 业务或校验错误经 `httpx.ErrorCtx` 返回错误信息（如无效链接、格式错误、自定义冲突等）。

### 2.2 短链解析并跳转

| 项目 | 说明 |
|------|------|
| **方法与路径** | `GET /:shortURL` |
| **路径参数** | `shortURL`：短链路径段（不含域名） |

**成功行为：** HTTP **302 Found**，`Location` 为数据库中的长链接 `lurl`。

**失败行为：** 通过 `httpx.ErrorCtx` 返回错误（逻辑层对「不存在」使用特定错误文案；过期返回「链接已过期」等）。

## 三、业务流程分析

### 3.1 转链流程（`ConverLogic.Conver`）

1. **连通性校验：** 对 `longURL` 发起 HTTP GET，非成功视为无效链接，终止。
2. **幂等（按长链）：** 计算 MD5，`FindOneByMd5`。若存在则直接返回已有记录的短链（**不再处理本次请求中的自定义短链与过期字段**，与「同一长链唯一映射」一致）。
3. **防套娃：** 取 URL 基底路径，若该路径已是已有 `surl`，则拒绝（视为已是短链）。
4. **生成 `surl`：**
   - **自定义：** 校验不在内存黑名单；`FindOneBySurl` 确认未被占用；否则报错。
   - **自动：** `Sequence.Next()` → Base62 → 若在黑名单则循环重新取号直至可用。
5. **过期时间：** 若传入 `expireAt`，按 RFC3339 解析；失败则报错。未传则为「数据库侧 NULL」表示永不过期。
6. **持久化：** `Insert` 写入 `lurl`、`md5`、`surl`、`expire_at`。
7. **布隆过滤器：** 插入成功后对该 `surl` 执行 `Filter.Add`，便于后续跳转前置过滤。
8. **响应：** 组装 `shortDomain + "/" + surl`，必要时带上 `expireAt` 字符串。

### 3.2 跳转流程（`ShowLogic.Show`）

1. **布隆过滤器：** `Filter.Exists(shortURL)`。若判定不存在，直接视为无效短链，**不访问数据库**。
2. **读映射：** `FindOneBySurl`，未找到则视为布隆假正例或数据不一致，按不存在处理。
3. **过期判定：** 若 `expire_at` 有效且当前时间晚于该时间，返回业务错误（链接已过期），不跳转。
4. **成功：** 返回长链；Handler 层发起 **302** 重定向。

### 3.3 启动预热（布隆过滤器）

服务启动时：`FindAllSurl` 查询 **`is_del = 0` 且 `surl` 非空，且 (`expire_at IS NULL` 或 `expire_at > NOW()`）** 的全部 `surl`，逐条 `Filter.Add`。用于在 Redis 中尽快对齐「当前仍有效的短链」集合（具体与 Redis 内已有状态叠加，取决于部署与持久化策略）。

## 四、缓存与并发相关问题在本项目中的解法

数据访问层使用 go-zero **`sqlc` + Redis**：按 `id`、`lurl`、`md5`、`surl` 等维度生成独立缓存键（前缀形如 `cache:shorturl:shortUrlMap:surl:` 等）。写路径以框架生成的 **`Insert` / `Update` / `Delete` 连带失效相关 Key** 为主（Cache-Aside 变体：**更新数据库并删除缓存**，读路径未命中再回填）。

### 4.1 缓存穿透（查询不存在的数据）

**含义：** 大量请求查询根本不存在的 key，缓存无法命中，压力打到数据库。

**本项目做法：**

- **跳转接口：** 先用 **Redis 布隆过滤器** 判断 `surl` 是否「可能存在」。过滤器判定不存在时，**直接结束**，不再执行 `FindOneBySurl`，从而避免对随机恶意路径的反复 DB 查询。
- **假正例：** 过滤器判定存在但 DB 无记录时，仍会查库一次；概率由布隆参数决定，属于可接受的兜底路径。
- **转链接口：** 不存在「短链 key」穿透问题；对**未出现过**的长链会先经过外网连通性校验，再查 MD5，恶意超长随机 URL 侧更多消耗在网络探测与单次查库，与本节经典「缓存空值」策略不同——当前代码**未**对「确定不存在」的 MD5 写入空值缓存，若需极致防护可另行引入短 TTL 空对象缓存或限流。

### 4.2 缓存击穿（热点 key 失效瞬间）

**含义：** 单个极高频 key 的缓存条目失效或被删除，瞬时大量请求同时miss，直击数据库。

**本项目做法：**

- 热点短链经首次命中后会长期受益于 **Redis 中单 key 缓存**；失效后的并发查询是否合并为单次回源，取决于 go-zero `CachedConn` 实现版本与配置，**业务代码未单独实现互斥锁 / singleflight**。
- **实务补强方向（未在当前仓库强制实现）：** 对极热点 key 使用 singleflight、或在缓存失效窗口使用短暂互斥锁 + 异步重建。

### 4.3 缓存雪崩（大量 key 同时失效）

**含义：** 批量缓存同一时间过期，导致 DB 请求陡增。

**本项目做法：**

- 缓存过期策略主要由框架与 Redis TTL 决定；**应用层未显式写入随机抖动 TTL**。
- 降低雪崩风险的常见手段（差异化 TTL、多级缓存、限流）可作为演进项单独评估。

### 4.4 与「过期时间」相关的语义

- **跳转前校验过期**在 **`ShowLogic` 内存中**基于 `ExpireAt` 与 `time.Now()` 完成，**晚于**缓存读取。因此即便 Redis 中仍为旧行，也会在应用层拦截已过期短链，避免「缓存命中即永久可跳转」的逻辑错误。
- 数据库与布隆过滤器：**标准布隆不支持删除**。短链创建时会 `Add`；过期后不会自动从布隆中移除，过期请求仍可能进入 DB 分支并在逻辑层被拒绝，属于有意权衡（避免误判不存在）。

### 4.5 小结表

| 问题 | 本项目主要手段 | 说明 |
|------|----------------|------|
| 穿透（无效短链） | Redis 布隆 + 启动预热 | 绝大部分随机路径在过滤器阶段拦截 |
| 击穿（热点） | Redis 缓存 | 依赖缓存扛量；未单独实现 singleflight |
| 雪崩 | 未单独处理 | 依赖 Redis / 运维 TTL 策略 |
| 一致性 | Cache-Aside（删缓存） + 跳转前过期校验 | 短链维度多 key 联动失效由 model 层生成代码维护 |
